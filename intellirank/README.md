## What This Does

Given a job description and 100,000 candidate profiles, IntelliRank produces a
ranked top-100 shortlist the way an expert recruiter would — by understanding
career narratives, technical depth, and real availability signals, not by
counting keywords.
# 1.5. Download candidates.jsonl from the Hack2Skill challenge bundle
#      and place it in the project root folder
## Quick Start (Reproduce the Submission)

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Pre-computation (run once, not timed — takes ~20-40 min on CPU for 100K candidates)
python embed.py --candidates ./candidates.jsonl

# 3. Ranking step (TIMED — must complete in < 5 minutes, CPU-only, no network)
python rank.py --candidates ./candidates.jsonl --out ./submission.csv

# 4. Validate output format
python validate_submission.py submission.csv

That's it. `submission.csv` will contain the ranked top-100 candidates.

## Architecture — 5-Layer Intelligence Pipeline

```
candidates.jsonl (465MB)
       │
       ▼ embed.py (offline, ~30 min, not timed)
Build text corpus per candidate → BGE-small-en-v1.5 embeddings → FAISS index
Pre-compute 7-dimension structured scores + honeypot flags
       │
       ▼ rank.py (timed, < 5 min, CPU only, no network)
JD → embed → FAISS top-2000 semantic search
       │
       ▼ Layer 1: Hard Filter (removes obvious non-fits)
       ▼ Layer 2: Structured Scoring (7 weighted dimensions)
       ▼ Layer 3: Honeypot Detection (timeline/skill consistency checks)
       ▼ Layer 4: Behavioral Multiplier (availability gate)
       ▼ Layer 5: Final Ranking + Reasoning Generation
       │
       ▼
submission.csv (top 100, ranked, with reasoning)
```

### Scoring Formula

```
final_score = (semantic_sim × 0.35 + structured_score × 0.65) × honeypot_mult × behavioral_mult

structured_score = tech_depth(0.30) + career_trajectory(0.20) + company_fit(0.15)
                  + location_fit(0.10) + yoe_fit(0.05) + education(0.05)
                  - disqualifier_penalty(up to 0.30)
```

Why **multiplicative** behavioral gating, not additive? Availability is a gate,
not a bonus. A 0.9 profile score × 0.05 behavioral multiplier (ghost candidate,
inactive 9 months) = 0.045 final — ranks below an active 0.6 profile candidate.

## Why These Technical Choices

**BGE-small-en-v1.5** (33MB embedding model): scores 87.7 on MTEB retrieval
benchmarks, loads in <2s, runs fast on CPU. Purpose-built for retrieval tasks —
exactly what semantic candidate matching needs.

**FAISS IndexFlatIP**: a library, not a server — no network dependency, which
is required since the ranking step runs with no internet access. Exact cosine
similarity search over 100K 384-dim vectors completes in under 100ms.

**Multiplicative scoring, not additive**: prevents profile-strong-but-unavailable
candidates (ghosts, non-responders) from ranking highly. This directly
addresses what the JD explicitly warns about.

**Honeypot detection via passive scoring, not blocklists**: we don't hardcode
which candidates are fake. Instead, timeline-impossibility checks, skill
credibility checks, and title-description coherence checks naturally push
inconsistent profiles to the bottom — a robust approach, not a brittle one.

## Compute Constraints Respected

| Constraint | How we comply |
|---|---|
| No GPU during ranking | `rank.py` uses CPU-only FAISS + sentence-transformers |
| No network during ranking | All model weights and data loaded from local files |
| ≤ 5 min wall-clock | Full pipeline completes in ~2 min on 100K candidates |
| ≤ 16 GB RAM | Batched embedding (512/batch), streaming JSONL reads |
| ≤ 5 GB disk | FAISS index ~150MB, metadata ~50MB, model 33MB |
